# Task 6: Implement Customer Email/Password Auth - Backend
Date: 2026-01-25

## Completed Work

### Customer Authentication Implementation
Implemented customer email/password authentication using Medusa v2's built-in auth system. The implementation follows Medusa v2 best practices and leverages the built-in auth routes.

### Authentication Flow
The customer authentication flow works as follows:

1. **Register Auth Identity**: `POST /auth/customer/emailpass/register`
   - Client sends `{ email, password }`
   - Returns registration JWT token

2. **Create Customer**: `POST /store/customers`
   - Client sends customer data with registration token in Authorization header
   - Creates customer profile linked to auth identity

3. **Login**: `POST /auth/customer/emailpass`
   - Client sends `{ email, password }`
   - Returns session JWT token

4. **Access Protected Routes**: Use token in `Authorization: Bearer {token}` header

5. **Password Reset**: `POST /auth/customer/emailpass/reset-password`
   - Triggers password reset flow via event system

### API Routes Created

#### POST /store/customers
- Creates new customer after registration token validation
- Validates input with Zod schema (email, first_name, last_name, phone, metadata)
- Checks for existing customer with same email
- Returns customer profile (without sensitive data)

#### GET /store/customers/me
- Protected route requiring customer authentication
- Returns authenticated customer's profile with addresses

#### POST /store/customers/me
- Protected route requiring customer authentication
- Updates customer profile (first_name, last_name, phone, metadata)
- Returns updated customer profile with addresses

### Middleware Configuration
- Protected `/store/customers/me` and `/store/customers/me/*` with customer authentication
- Configured `/store/customers` POST to accept registration tokens (allowUnregistered: true)
- Uses both `bearer` and `session` authentication methods

### Input Validation
- Used Zod schemas for request validation
- CreateCustomerSchema: email (valid format), first_name (required), last_name (required), phone (optional), metadata (optional)
- UpdateCustomerSchema: first_name, last_name, phone (nullable), metadata (all optional)

# Task 10: Implement Phone OTP Auth - Backend
Date: 2026-01-25

## Completed Work

### Phone OTP Authentication Implementation
Implemented phone-based OTP (One-Time Password) authentication system using Twilio for SMS delivery. The implementation provides an alternative authentication method for customers who prefer phone-based login.

### Authentication Flow
The phone OTP authentication flow works as follows:

1. **Request OTP**: `POST /store/auth/otp/send`
   - Client sends `{ phone: "+96512345678" }`
   - Server generates 6-digit OTP
   - Server stores OTP with 5-minute expiry
   - Server sends OTP via Twilio SMS
   - Returns success/failure response

2. **Verify OTP**: `POST /store/auth/otp/verify`
   - Client sends `{ phone: "+96512345678", code: "123456" }`
   - Server validates OTP code
   - On success, creates customer if not exists or updates existing customer
   - Returns customer profile with verification status

### OTP Service Features
- **OTP Generation**: Generates cryptographically random 6-digit codes
- **OTP Storage**: In-memory store (development) with TTL support
- **Phone Normalization**: Supports international formats (+, 00, with/without country code)
- **Rate Limiting**: Max 5 OTP requests per phone per hour
- **Attempt Limiting**: Max 3 verification attempts per OTP
- **Expiry**: OTPs expire after 5 minutes
- **SMS Integration**: Uses existing Twilio SMS notification service

### API Routes Created

#### POST /store/auth/otp/send
- Validates phone number format
- Checks rate limits (5 requests/hour)
- Generates and stores OTP
- Sends OTP via Twilio SMS
- Returns 200 on success, 429 on rate limit, 400 on invalid phone

#### POST /store/auth/otp/verify
- Validates phone and code format
- Verifies OTP code against stored value
- Handles expiry, invalid codes, max attempts
- Creates new customer or updates existing
- Returns customer profile with `is_new` flag
- Returns 200 on success, 400 on invalid, 410 on expired, 429 on max attempts

### Input Validation
- Used Zod schemas for request validation
- SendOTPSchema: phone (8-20 chars, valid format)
- VerifyOTPSchema: phone (8-20 chars), code (exactly 6 digits)

### OTP Store Interface
```typescript
interface OTPStore {
  set(key, value, ttl): Promise<void>
  get(key): Promise<OTPRecord | null>
  delete(key): Promise<void>
  incrementAttempts(key): Promise<number>
  getAttempts(key): Promise<number>
}
```

## API Usage Examples

### Request OTP
```bash
curl -X POST http://localhost:9000/store/auth/otp/send \
  -H "Content-Type: application/json" \
  -d '{"phone": "+96512345678"}'
```

Response:
```json
{
  "success": true,
  "message": "OTP sent successfully. Please check your phone."
}
```

### Verify OTP
```bash
curl -X POST http://localhost:9000/store/auth/otp/verify \
  -H "Content-Type: application/json" \
  -d '{"phone": "+96512345678", "code": "123456"}'
```

Response:
```json
{
  "verified": true,
  "message": "Phone number verified successfully",
  "customer": {
    "id": "cust_123",
    "email": "phone_96512345678@placeholder.local",
    "phone": "+96512345678",
    "first_name": "",
    "last_name": "",
    "has_account": true,
    "is_new": true
  }
}
```

## Security Considerations
- OTPs are stored in memory (use Redis for production)
- Rate limiting prevents brute force attacks
- Max attempts limit prevents code guessing
- Phone numbers are normalized for consistent storage
- Placeholder emails used for customers without email
